# Code review process template

This document describes a repeatable review process that can be reused
across projects and packages, including AI-assisted follow-up reviews.
It is intended to be language- and domain-agnostic.

## Goals

- Prioritise **performance**, then **maintainability** (including
  testability and clarity of invariants).
- Keep the public API consistent and predictable for callers.
- Capture findings in versioned review documents under
  `docs/reviews/` with stable issue identifiers.

## Review artefacts

Each review run produces one or more markdown files:

- `docs/reviews/rev-NN.md` for the findings of a specific run
  (e.g. `rev-01.md`).
- `docs/reviews/rev-0000.md` (this file) describing the process that
  later reviews should follow.

Within each `rev-NN` document:

- Issues are numbered `REV-NN-{k}` (for example, `REV-01-3`).
- The top of the file contains a **summary table** listing all issues,
  their category, severity, and a short description.
- Substantial issues get their own section.
- Minor issues may be grouped into sections like
  "Consistency issues", "Performance issues", "Suspected defects",
  or "Test gaps".

## High-level procedure for an AI reviewer

1. **Establish context**
   - Confirm the working directory and repository.
   - Identify the component under review (package, module, service,
     or subsystem).
   - Read any existing review documents (`rev-*.md`) to understand
     prior findings and open issues.

2. **Inventory the public API**
   - List all exported identifiers in the target package:
     - Constants, types, variables, and functions.
   - Classify them into coarse categories:
     - Construction / builders.
     - Persistence / layout helpers.
     - Query and proof APIs.
     - Sizing and integration helpers.
   - Cross-check API comments (`doc.go` and per-symbol comments) for
     consistency with the actual signatures and behaviour.

3. **Static implementation review**
   - For each exported function and type, inspect the implementation:
     - Check for off-by-one errors and incorrect bounds checks.
     - Confirm that panic paths are justified (and only used where
       performance dictates and invariants are internal).
     - Validate how errors are surfaced and whether they can be
       distinguished by callers.
   - Pay special attention to invariants that affect safety or
     performance, such as:
     - Assumptions about key ordering and key bit traversal.
     - Assumptions about fixed buffer sizes and record layouts.
     - Encoding/decoding of on-disk state (e.g. frontier blocks).

4. **Performance considerations**
   - Identify which operations are on the hot path for this project
     (for example, request handling, indexing, or critical data
     structures).
   - For these functions and code paths:
     - Look for unnecessary allocations (for example, per-call
       slices or maps).
     - Check for redundant work (duplicate parsing, hashing,
       conversions, or I/O).
     - Ensure that any additional safety checks are either
       amortised or applied only at boundaries (for example,
       decode time, configuration load, or startup).
   - It is generally acceptable for cold-path functions (such as
     configuration validation or decode-time checks) to perform
     extra validation if it improves robustness.

5. **Test and tooling review**
   - Locate and read the existing tests for the package.
   - Identify which invariants are explicitly asserted in tests and
     which are only implicitly relied upon.
   - Where possible, run the tests for the relevant module(s).
     - If the package is not part of a standalone module, ensure that
       you understand how its tests are intended to be run (for
       example, as part of a larger workspace).
   - Look for missing tests in the following areas:
     - Edge cases for encoders/decoders and sizing helpers.
     - Error paths (invalid inputs, out-of-range values).
     - Negative tests for proof verification (tampering, mismatch).

6. **Identify and classify issues**
   - For each potential issue, decide whether it should be recorded as
     a formal finding:
     - **High severity**: potential correctness bugs, panics from
       untrusted input, or significant performance regressions.
     - **Medium severity**: confusing or fragile APIs, test gaps around
       important invariants, or inconsistent contracts between
       helpers.
     - **Low severity**: minor naming inconsistencies, redundant code,
       or non-critical documentation gaps.
   - Assign an ID of the form `REV-NN-{k}` and choose a category such as
     "Robustness", "API semantics", "Invariants", "Performance", or
     "Test coverage".

7. **Update the review document (`rev-NN.md`)**
   - Create or update the markdown file for the current review run.
   - At the top, add or update the issue summary table.
   - For substantial issues, add dedicated sections that include:
     - A clear statement of the finding.
     - The impact on correctness, performance, or maintainability.
     - Concrete recommendations or options for remediation.
   - Group minor issues into thematic sections (for example,
     "Consistency issues"), listing each issue with its ID.
   - Keep line lengths at or below 80 characters for headings and
     paragraphs where practical, without breaking words. Tables may
     exceed the limit if necessary.

8. **Cross-reference and follow-up**
   - When a new review supersedes or revisits an old issue, reference
     the original issue ID(s) and describe whether they are now:
     - Resolved.
     - Partially addressed.
     - Still open, with any new context.
   - If code changes are made in response to a review, consider adding
     notes in the corresponding `rev-NN.md` explaining which issues
     they address and where the changes live in the tree.

## Adding new review runs

For a new review run (for example, `REV-02`):

1. Create `docs/reviews/rev-02.md` (or the equivalent path for your
   project) following the structure of the previous review document
   (for example, `rev-01.md`):
   - Header with date, scope, and priorities.
   - Issue summary table.
   - Per-issue sections.
2. Reuse and adapt the procedure in this document, focusing on the
   relevant scopes:
   - Additional components (for example, cross-package integration,
     new services, or new subsystems).
   - New features or substantial refactors.
3. Keep issue IDs unique within the review run and monotonic for easy
   reference (for example, `REV-02-1`, `REV-02-2`, ...).

## Expectations for future AI-driven reviews

- Apply this process conservatively: prefer fewer, well-supported
  findings over many speculative ones.
- When in doubt about a potential issue, describe it explicitly as a
  hypothesis and state what additional evidence (tests, benchmarks,
  design docs) would be needed to confirm it.
- Always respect the stated priority ordering:
  - Do not propose changes that materially hurt hot-path performance
    unless they address a clear correctness bug.
  - Favour changes that improve robustness at boundaries (decode,
    configuration checks) and that are easy to test.
- Keep review documents concise but sufficiently detailed that a human
  maintainer can act on them without needing to reconstruct all of the
  reasoning from scratch.
